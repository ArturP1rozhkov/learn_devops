## Задача 2:

Выберите один из вариантов платформы в зависимости от задачи. Здесь нет однозначно верного ответа так как все зависит от конкретных условий: финансирование, компетенции специалистов, удобство использования, надежность, требования ИБ и законодательства, фазы луны.

Тип платформы:

- физические сервера;
- паравиртуализация;
- виртуализация уровня ОС;

Задачи:

- высоконагруженная база данных MySql, критичная к отказу;
- различные web-приложения;
- Windows-системы для использования бухгалтерским отделом;
- системы, выполняющие высокопроизводительные расчёты на GPU.

Объясните критерии выбора платформы в каждом случае.

## Ответы:

### Высоконагруженная MySQL, критичная к отказу: наиболее оптимальный выбор - физический сервер



- MySQL критична к скорости дисковых операций. Любая прослойка виртуализации снижает скорость операций. 
    
- На виртуальной машине другие ВМ на том же хосте могут «съедать» ресурсы CPU и памяти 
    
- физический сервер с RAID и резервным питанием более надёжен (гипервизор добавляет точку отказа)
    
- MySQL на OLTP-нагрузке интенсивно использует  ядра для обработки запросов, и на физическом сервере тактовая частота не делится ни с кем
    

### Различные веб-приложения: выбор - виртуализация уровня ОС (контейнеры)


- каждое приложение в своём контейнере со своими зависимостями (Python 3.8 для одного, Python 3.11 для другого)
    
- контейнер запускается за секунды, в отличие от отдельных виртуальных машин
    
- контейнеры эффективно расходуют ресурсы не тратя память на запуск множества отдельных операционных систем 
    
- легко горизонтально масштабировать под нагрузкой, то есть при необходимости поднять дополнительные контейнеры
    
- **Микросервисная архитектура** — каждый сервис в отдельном контейнере
    
    Однако есть ограничение: все контейнеры используют **одно ядро ОС хоста**, поэтому если нужны разные ОС (например, Windows + Linux) — контейнеры не подойдут.
    

---

###  Windows-системы для бухгалтерии: выбор - паравиртуализация (гипервизор)


- Возможность использования разных ОС на одном железе — пользователи (бухгалтеры) Windows, а серверная инфраструктура на Linux. Гипервизор (Hyper-V, VMware, KVM) позволяет запустить Windows-ВМ на Linux-хосте
    
- Требования безопасности — каждая ВМ изолирована на своем ядре процессора, поэтому компрометация одной ВМ не затронет другие. 
    
- Можно мгновенно откатить ВМ к предыдущему состоянию, что удобно при обновлениях
    
- Централизованное управление несколькими рабочими местами через гипервизор (VDI)
  

---
### Высокопроизводительные расчёты на GPU: выбор - физический сервер (или паравиртуализация с прямым доступом к GPU)**


- Стандартная виртуализация с GPU даёт значительные потери производительности
    
- Виртуализация с GPU pass-through или прямым доступом к графическому процессору позволяет пробросить физический GPU напрямую в ВМ (1 GPU = 1 ВМ), практически без потерь производительности. Это паравиртуализация, но с прямым доступом к GPU
    
- GPU передаёт огромные объёмы данных через PCIe. Любая виртуализация шины создаёт узкое место с потерей производительности



## Задача 3
   
   Выберите подходящую систему управления виртуализацией для предложенного сценария. Опишите ваш выбор.
   
   Сценарии:
   
   1. 100 виртуальных машин на базе Linux и Windows, общие задачи, нет особых требований. Преимущественно Windows based-инфраструктура, требуется реализация программных балансировщиков нагрузки, репликации данных и автоматизированного механизма создания резервных копий.
   2. Требуется наиболее производительное бесплатное open source-решение для виртуализации небольшой (20-30 серверов) инфраструктуры на базе Linux и Windows виртуальных машин.
   3. Необходимо бесплатное, максимально совместимое и производительное решение для виртуализации Windows-инфраструктуры.
   4. Необходимо рабочее окружение для тестирования программного продукта на нескольких дистрибутивах Linux.

## Ответ
1. **Proxmox VE**. 
	- **Единый веб-интерфейс** для управления всеми 100 виртуальных машин (ВМ) — не нужно заходить на каждый узел отдельно
	   
	- **KVM + LXC (Linux Containers) в одной системе**.  Windows-машины через KVM (полная виртуализация), однотипные Linux-сервисы через LXC-контейнеры (экономия ресурсов без эмуляции железа)
	   
	- **Кластеризация и миграция**.  ВМ можно переносить между физическими узлами без остановки
	   
	- **Встроенное резервное копирование** через Proxmox Backup Server, который включён в экосистему бесплатно
	   
	- **Масштаб** сценария: 100 ВМ это типичная нагрузка для Proxmox, он спроектирован именно под такие задачи

2. **Proxmox VE**. Акцент на его бесплатности: Для сравнения можно взять также **oVirt** и **XenServer**. Они также бесплатны (лицензия Apache). Все они имеют управление через веб-интерфейс (Xen Server также управляется через CLI), позволяют управлять виртуальными машинами с OS Windows, но в отличие от **Proxmox** - **oVirt** и **XenServer** не имеют встроенного LXC, а для использования кластера машин требуется дополнительная настройка (в Proxmox кластер встроен), их порог входа значительно выше чем для Proxmox и требует дополнительной квалификации от инженера администратора.

3. **Microsoft Hyper-V Server (Free)**. Ключевое  — **максимальная совместимость** с Windows. 
	- **Hyper-V Server** — это бесплатный standalone-продукт Microsoft, только гипервизор без GUI Windows Server
	    
	- **Встроены во все современные версии Windows** — гостевые ОС работают без ограничений, костылей и ручной установки драйверов
	    
	- **VHD/VHDX (Virtual Hard Disk - старый и новый форматы файлов виртуальных дисков от Microsoft) - диски** совместимы между Hyper-V, Azure и WSL2 — максимальная портируемость без дополнительной конвертации, поскольку все диски витртуальных машин хранятся в системе в виде файлов .vhdx (единый формат Microsoft), которые при копировании например в облачную инфраструктуру принимаются Azure напрямую, в отличие от аналогчного варианта в VMvare - формат .vmdk, при портировании файлов необходима дополнительная конвертация.
	    
	- **Поддержка Secure Boot, UEFI, современных чипсетов** именно для Windows-машин
	    
	- **Полная автоматизация** управления через родные Windows-инструменты PowerShell и WinRM 

4. **VirtualBox (или Vagrant + VirtualBox)**. Это desktop-сценарий, где требования  — не производительность, а **удобство и скорость разворачивания**.
	- **VirtualBox бесплатный и кроссплатформеный**, то есть работает на Windows, Linux, macOS, то есть каждый разработчик использует одни и те же образы на своей машине
	    
	- Можно заморозить чистое состояние любого дистрибутива с помощью **снапшотов** и откатываться к нему после каждого теста
	    
	- Код можно редактировать на хосте и тестировать внутри ВМ
	    
	- **Vagrant + VirtualBox** — один Vagrantfile описывает окружение, командой `vagrant up` поднимаются все нужные дистрибутивы автоматически
	


## Задача 4
Опишите возможные проблемы и недостатки гетерогенной среды виртуализации (использования нескольких систем управления виртуализацией одновременно) и что необходимо сделать для минимизации этих рисков и проблем. Если бы у вас был выбор, создавали бы вы гетерогенную среду или нет?

## Ответ:
**Гетерогенная среда** — это когда в одной инфраструктуре одновременно работают **несколько разных платформ виртуализации**: например, Proxmox + Hyper-V + VMware. Каждая платформа управляется своими инструментами, использует свои форматы дисков и протоколы.

### Основные проблемы и недостатки
   
#### 1. Фрагментация управления
Для каждой платформы — свой отдельный инструмент администрирования. Нет единой консоли, которая видела бы все ВМ сразу. Администратор вынужден переключаться между vSphere, Hyper-V Manager и Proxmox Web UI, что увеличивает время реакции на инциденты и вероятность ошибок.
   
#### 2. Несовместимость форматов
Диски ВМ в разных форматах: `.vmdk` (VMware), `.vhdx` (Hyper-V), `.qcow2` (KVM/Proxmox). Миграция ВМ между платформами очень сложный и рискованный процесс: файлы дисков требует конвертации — это время, риск потери данных и простой сервиса.
   
#### 3. Проблемы с поддержкой и лицензированием
Вендоры ПО поддерживают свои продукты только в сертифицированных ими средах виртуализации. Например, Microsoft может отказать в поддержке Windows Server, если он запущен на гипервизоре, который не сертифицирован для данной ОС. Кроме того, каждая коммерческая платформа (VMware, Hyper-V) требует отдельной лицензии.
   
#### 4. Требование компетенций по каждой платформе
Каждая система виртуализации — отдельная специализация. Администратор VMware не знает нюансов Proxmox, и наоборот. Либо нужны разные специалисты (дорого), либо один человек поверхностно знает всё (опасно).
   
#### 5. Сложность мониторинга
У каждой платформы своя система метрик и алертов. Собрать единую картину производительности и доступности инфраструктуры крайне сложно — появляются **«слепые зоны»**, где инциденты остаются незамеченными.

#### 6. Проблемы безопасности
Разные платформы — разные политики безопасности, разные векторы атак, разные циклы обновлений. Сложно обеспечить единый аудит событий и соответствие требованиям ИБ.
   
#### 7. VM Sprawl (виртуальное разрастание) усугубляется
В гетерогенной среде неконтролируемое разрастание ВМ происходит быстрее: каждая команда плодит машины на «своей» платформе, забывает удалять, бросают, теряют владельцев и никто не видит общей картины.


#### В итоге:
- **Расходование ресурсов** — железо занято бесполезными машинами, новым сервисам не хватает мощности
  
- **Угроза безопасности** — брошенные ВМ не обновляются, становятся дырой в безопасности
  
- **Потеря контроля** — администратор боится удалить ВМ, не зная, нужна ли она кому-то

#### Варианты минимизации упомянутых рисков:
## Как бороться

- Вести актуальный **реестр всех ВМ** с указанием владельца и назначения
    
- Автоматически **отключать ВМ**, которые не использовались N дней (требования регламента предприятия, владельца ресурсов, требует отдельной разработки отраслевой нормативной базы)
    
- Ввести **процесс согласования** при создании новой ВМ — с указанием срока жизни (опять же должно быть предусмотрено отраслевой нормативной базой)
    
- Использовать инструменты мониторинга (Zabbix, vRealize), которые сами выявляют простаивающие машины

### Создавал бы я гетерогенную среду, если бы у меня был выбор?
   
#### Нет. 
   
   **Гетерогенная среда** — это почти всегда результат **исторических решений**, накопленных проблем, наслоение решений разных команд администраторов, а не осознанного проектирования. Например компания купила VMware, потом добавила Hyper-V для Windows, потом кто-то поднял Proxmox для тестов — и вот уже три платформы. 
   
   При проектировании с нуля разумный выбор — **одна платформа**, покрывающая все сценарии:
   
   - **Proxmox** — если бюджет ограничен и нужна смешанная Linux/Windows среда
       
   - **Hyper-V** — если инфраструктура преимущественно Windows
       
   - **VMware vSphere** — если есть бюджет и нужен enterprise-уровень поддержки (или отечественные решения типа "Горизонт ВМ" в случае импортозамещения)
       
Единственный обоснованный вариант для гетерогенной среды — **миграционный период**, когда ты целенаправленно переводишь инфраструктуру с одной платформы на другую. Тогда временное сосуществование двух систем оправдано и запланировано. Но это не должно быть бесконтрольным и хаотичным и быть завязанным на "незаменимого" администратора.
